## @核心原则

**Karpathy 行为准则融合**（v26 新增，来源于 Andrej Karpathy LLM 编码注意事项）：
- **假设透明**：ingest 前声明假设，不确定时主动询问，不要猜测
- **最小代码**：只做必要的变更，不做推测性功能
- **精准触及**：每个变更都可追溯到用户请求，阶段产出与请求直接对应
- **目标驱动**：每个阶段有明确的 verify 检查点

**RAG vs LLM Wiki**：ingest 不是"给文档建索引"，而是"把知识编译进网络"。

**绝对红线**：
- `sources/` 目录下的原始文件**只读不删**
- 永不生成占位符内容（`TODO`、`待补充`、`（略）`）
- 量刑档次三档/四档必须全部提取
- **法律准确性**：涉及具体法条内容（数额、日期、适用范围、罪名构成等）必须引用原文，禁止主观推测；**不确定时必须向用户核验**，严禁编造

**执行纪律**：
- Agent 报告不可信 → 必须用脚本验证
- 内容质量门控：正文 < 500 字符必须立即复核
- 批量变更后立即 commit
- **法律条文对比**：涉及多部法律/解释比较时，必须先读取原文再下结论
- **Clippings 文件**：处理完成后必须移动到 `sources/网络文章/`（详见阶段四补充）
- **skill 版本控制**：修改 SKILL.md 前必须先 git commit 当前版本；每次版本更新后立即 commit（防止回滚丢失历史）
- **目录扫描后穷尽验证**：处理完目录内文件后，**必须重新扫描目录确认文件数量**，不得凭记忆下结论；数量是硬指标，说了处理 N 个文件但实际少了 1 个就是没闭环
- **声称状态前必须验证**：说"目录已清空/有X个文件/已处理完"之前，**必须先输出工具扫描结果作为证据**，不允许零证据输出结论
- **完成交付验证标准**：所有 ingest 操作完成后，必须输出验证数据（文件数量、ls 输出、行数统计），不说"处理完了"只说"验证结果：X个文件，全部处理"
- **修改 SKILL.md 前验证**：修改 skill 文件前，必须先执行 `git status && git log -1` 确认当前版本已 commit，再进行修改
- **Sub-agent 结果主会话验证**：Agent 报告完成不等于完成，主会话必须用脚本重新验证实际文件数量，不接受 Agent 报告作为最终结论

**数字/日期信息处理原则**（v27 新增）：
- 涉及**数额标准**（如受贿罪3万/20万/300万）、**日期**（生效日期、修改日期）、**期限**（羁押期限、追诉时效）等关键数据时，**必须标注来源文件**
- 同一数据在不同页面出现时，必须一致；发现矛盾时在相关页面标注 ⚠️
- 法律条文中的数字/日期不得主观推测或估算，必须引用原文

**法条内容处理原则（两步验证）**（v27 新增）：
1. **第一步**：提取法条原文时，确保内容与原文一致（逐字核对关键条款）
2. **第二步**：写入 wiki 前，验证法条编号、条款顺序、量刑幅度是否与源文件匹配
- 涉及数额、日期、生效时间等关键数据，标注 `① 来源：xxx`
- 存在多种解释或争议的内容，标注 `⚠️ 争议点：...`

**Anti-Hallucination Mandate（三色断言）**（v38 新增，来源于 ARS）：
每个法条引用后必须标注验证状态：
- ✅ **VERIFIED**：已核对原文，确认无误
- ❌ **NOT_FOUND**：原文未找到该表述
- ⚠️ **MISMATCH**：存在矛盾，需进一步核验

**Material Passport 格式（v38 新增）**：
sources 字段可选增强为结构化格式：
```yaml
sources:
  - path: sources/xxx.md
    date: YYYY-MM-DD
    confidence: ⭐⭐⭐⭐  # 知识可信度等级
    extracted: YYYY-MM-DD
    extractor: Agent/主会话
```

## @自动模式：Clippings 检测触发

**@触发机制**：
当 skill 被调用时，自动检查 vault 根目录下的 `Clippings/` 目录是否存在 .md 文件。

**@检测命令**：
```bash
ls D:/AI\ agent/tkk-library/Clippings/*.md 2>/dev/null | wc -l
```

**@行为逻辑**：

| 情况 | 行为 |
|------|------|
| Clippings 为空 | 继续正常操作（用户指定的 ingest/query/lint/purge） |
| Clippings 有文件 | **立即自动执行 ingest 流程**（阶段一→阶段五），处理完成后自动归档到 sources/网络文章/，无需询问 |

**@自动 ingest 执行流程**：
```
检测到 D:/AI\ agent/tkk-library/Clippings/*.md
    ↓
自动执行 ingest（阶段一→阶段五）
    ↓
阶段四完成后：询问用户是否归档到 sources/网络文章/
    ↓
阶段五：生成完成报告，询问用户下一步
```

**@用户干预时机**：
- 可以在任意时刻说"取消"或"停止"终止自动流程
- 自动流程完成后，默认进入等待状态，可继续执行其他操作
- 是否将 Clippings 文件移动到 sources/网络文章/，由用户决定（不强制）

**@为什么自动执行**：
Clippings/ 是用户主动放入的待整理资料，放入即表示需要处理——无需二次确认，直接开始整理。

```
┌──────────────────────────────────────────────────────────────┐
│                        一键整理模式                          │
│                                                              │
│   ingest触发 → 阶段一 → 阶段二 → 阶段三 → 阶段四 → 阶段五    │

│                 扫描源  Agent提取 脚本核验  整合入库   完成报告 │

阶段一（必须先执行）：扫描源文件 + 去重检查
阶段二：内容提取（分层，不是整文件）
阶段三：数量核验 + 内容质量检查（必须通过才能继续）
阶段四：整合入库（6步骤，缺一不可）
阶段五：完成报告 + 询问用户下一步
```

**@阶段级 Verify 检查点**（Karpathy Goal-Driven Execution 融合）：
```
ingest触发 → 
  假设声明：用户意图/文件类型/命名规则是否有疑义？
  ↓
阶段一 → verify: 源文件清单完整，无重复
阶段二 → verify: 每文件产出内容 > 500 字符
阶段三 → verify: 数量精确匹配，无占位符
阶段四 → verify: 6步骤全部执行（不只是 qmd embed）
阶段五 → verify: log.md 记录完整 + 用户确认

⚠️ 关键：阶段二/三完成后，必须重新扫描源目录验证文件数量
      说"处理了 N 个文件"但实际少 1 个 = 没闭环（红线一）
```

## @命令速查

```bash
# === ingest 常用命令 ===

# 阶段一：扫描源文件
ls sources/xxx/*.md | wc -l                    # 统计文件数
ls sources/xxx/*.md | grep -E '\([0-9]+\)$'   # 检测重复后缀

# 阶段三：精确比对（发现遗漏）
for f in sources/xxx/*.md; do
  name=$(basename "$f" .md)
  base_clean=$(echo "$name" | sed 's/（[0-9]*年[修正修订]*）//g')
  [ -f "wiki/concept_${name}.md" ] && continue
  [ -f "wiki/concept_${base_clean}.md" ] && continue
  echo "$name"
done

# 阶段三：占位符扫描
grep -r "待补充\|见原文\|（略）" wiki/summaries/*.md -l

# 阶段三：文件大小检查
for f in wiki/concepts/*.md; do
  size=$(wc -c < "$f")
  [ "$size" -lt 5000 ] && echo "⚠️ 可疑：$size bytes: $(basename "$f")"
done

# 阶段四：整合入库
qmd update && qmd embed

# === lint 常用命令 ===

# 断链检查
grep -r "\[\[" wiki/ | grep -v "\[\[.*\]\]"

# 源文件完整性验证（完整脚本见 lint 章节）
grep -r "^source:" wiki/summaries/*.md | sed 's/.*\///' | cut -d: -f1 | sort -u > /tmp/summary_sources.txt
ls sources/*.md sources/网络文章/*.md 2>/dev/null | sed 's|.*/||' | sort -u > /tmp/all_sources.txt
diff /tmp/summary_sources.txt /tmp/all_sources.txt | grep "^<" | sed 's/^< //' | while read f; do [ -f "sources/$f" ] || echo "⚠️ $f"; done

# === purge 命令 ===

# 查询源文件对应的wiki页面
grep "源文件名" sources/INDEX.md

# 删除前确认
ls wiki/summaries/源文件名*.md wiki/concepts/源文件名*.md
```

## 文件类型与目标目录

| 文件类型 | 目标目录 | type | 必需字段 |
|----------|----------|------|----------|
| 律师业务指引 | `wiki/summaries/` | `summary` | `category` |
| 法律法规 | `wiki/concepts/` | `concept` | `tags`含法律部门 |
| 司法解释原文 | `wiki/entities/` | `entity` | `tags`含解释文号 |
| 案例分析 | `wiki/syntheses/` | `synthesis` | `tags`含罪名 |
| 法律修改决定 | `wiki/concepts/` | `concept` | — |
| 综合分析 | `wiki/syntheses/` | `synthesis` | — |

**判断方法**：文件名含"指引"→ summaries；含"解释"/"规定"→ entities；含"案例"/"指导案例"→ syntheses

**summary 命名规范**：
| 来源类型 | 命名前缀 | 示例 |
|----------|----------|------|
| 最高检指导性案例 | `SPP_GuideCase_` | `SPP_GuideCase_01_...md` |
| 最高法入库案例/法答网 | `Court_FaDaWang_` | `Court_FaDaWang_01法答网精选答问第一批.md` |
| 律师业务指引 | `LawyerGuide_` | `LawyerGuide_...md` |
| 其他来源 | `Summary_` | `Summary_...md` |

> ⚠️ **命名一致性要求**：同一来源系列的文件必须使用相同的命名前缀，不得混用。批量处理前先检查同系列已有文件的命名前缀，沿用而非另起炉灶。

## 整理级别（五层划分）

用户可通过指定整理级别调整处理深度。**第3层（详细级）为默认级别**，未指定时自动应用。

整理级别是**五维向量**：
- **粒度维度**（横向）：极简→完备
- **置信度维度**（正交）：内容确定性
- **演化状态维度**（正交）：与前序版本的关系
- **作者意图保留度**（正交）：忠实于原文的程度

---

### 三维正交维度说明

#### 置信度

| 标签 | 含义 | 适用场景 | 处理方式 |
|------|------|----------|----------|
| 〔确定〕 | 内容与原文一致，无争议 | 法条原文、司法解释原文 | 直接写入 wiki |
| 〔存疑〕 | 存在多种解释或争议点 | 学理分歧、实践争议 | 写入 wiki + 标注〔存疑〕 |
| 〔待核实〕 | 无法确认，需进一步查证 | 转述内容、来源不明 | 写入 wiki + 标注〔待核实〕+ 同步 log.md |

**Layered Confidence Rejection**（v39 新增）：内容提取时按置信度分层拦截，不同级别的内容走不同处理路径，极低置信度内容（高度不确定/逻辑跳跃）拒绝写入 wiki，只存入 log.md 待验证。

#### 演化状态

| 标签 | 含义 | 示例 |
|------|------|------|
| 〔替代〕 | 完全替代旧版本 | 刑法修正案（十一）替代刑法修正案（十） |
| 〔优先适用〕 | 冲突条款以新版本为准，不冲突条款仍有效 | 法释〔2026〕6号（二）vs 法释〔2016〕9号 |
| 〔补充〕 | 在旧版本基础上增加新规定 | 补充规定 |
| 〔解释〕 | 对旧版本的细化解释 | 理解与适用 |
| 〔征求意见稿〕 | 尚未正式施行 | 草案 |

演化状态字段格式：
```yaml
演化状态:
  类型: 〔优先适用〕  # 必填
  说明: "与其冲突以本解释为准，不冲突条款仍有效"  # 选填
  冲突条款: "第一条至第四条（单位受贿罪等数额标准）"  # 选填
  不冲突条款: "受贿罪、贪污罪、挪用公款罪个人犯罪数额标准"  # 选填
```

#### 作者意图保留度

| 级别 | 保留度 | 说明 |
|------|--------|------|
| 第1层 | 100% | 逐字复制，不加解读 |
| 第2层 | 80% | 保留核心原文表述，加简短提炼 |
| 第3层 | 60% | 转译+提炼，增加自己的理解框架 |
| 第4层 | 40% | 重构+关联，以我为主 |
| 第5层 | 0% | 纯自己的认知框架，用于讲课/写文章 |

| 级别 | 名称 | 处理粒度 | 置信度 | 演化状态 | 意图保留 | 适用场景 |
|------|------|----------|--------|----------|----------|----------|

**指定整理级别的方式**：
- "整理xxx，第3层" / "整理xxx，详细级" / "整理xxx，默认级" → 第3层

| 元信息 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 完整目录 | - | ✓ | ✓ | ✓ | ✓ |

## ingest：消化新资料

### ⚠️ 首次扫描与末次扫描铁律（v31 新增）

**问题根因**：第一次只扫描了"当时存在的文件"，后来新增的文件永远发现不了。

**铁律**：ingest 操作分两个时间点必须扫描源目录：
1. **首次扫描**（处理前）：`ls sources/xxx/*.md | wc -l` → 记录为基准数量
2. **末次扫描**（处理后）：`ls sources/xxx/*.md | wc -l` → 再次执行，对比是否一致

**末次扫描的作用**：
- 如果两次数量一致 → 处理期间无新增文件，验证通过
- 如果两次数量不一致 → 发现了处理期间新增的文件，必须继续处理遗漏的文件
- **禁止跳过末次扫描**：即使"确信没有新增文件"也必须执行，这是验证不是可选步骤

**执行顺序**：
```
首次扫描 → 记录: "源目录: X个文件"
    ↓
处理文件（阶段一至五）
    ↓
末次扫描 → 对比: "源目录: X个文件" vs "处理: X个文件"
    ↓
数量一致 → 验证通过，输出"验证结果：源目录X个，处理X个，全部完成"
数量不一致 → 主会话接管，继续处理遗漏文件
```

### 阶段一：源文件清单分析 + 去重检查

**@步骤**：

1. `ls sources/xxx/*.md | wc -l` 统计总数
2. 识别文件类型分布（律师指引/法律法规/司法解释/案例）
3. 识别命名模式（日期格式、前缀规律）
4. **@强制：检查同系列已有 wiki 文件的命名前缀**
   ```bash
   # 批量处理前，先检查同系列已有文件的命名前缀，沿用而非另起炉灶
   ls wiki/summaries/ | grep -E "关键词" | head -3  # 如 Court_/SPP_GuideCase_/LawyerGuide_
   ```
5. 识别超大文件（>5000行）和普通文件

**@分组规划**：

| 规模 | 策略 |
|------|------|
| < 20个 | 单 Agent 串行 |
| 20-100个 | 按法律部门分组，≤30个/组 |
| > 100个 | 分层处理，每组≤30个且≤6000行 |
| 超大文件（>5000行） | **单独处理，见下方专项规则** |

**@超大文件专项处理规则（>5000行）**：

1. **识别文件结构**：
   ```bash
   grep -n "^# " sources/超大型文件.md | wc -l   # 统计章节数
   grep -n "^## " sources/超大型文件.md | wc -l  # 统计子章节数
   ```
2. **判断类型**：
   - **合订本**（如司法解释全书含189个独立解释）→ 按 `# 标题` 拆分为多个 entity，每批处理 10-20 个章节
   - **单一本**（如某罪的完整释义）→ 按 `## 标题` 拆分为多个 section，分批处理
3. **批次规划**：每批处理不超过 20 个 `# 章节`，完成后验证（数量核验 + 内容抽检），再处理下一批
4. **提取规则**：每个 `# 标题` 对应一个 `entity_*.md` 文件，逐条提取条款内容
5. **目标目录**：`wiki/entities/`（司法解释全书属于 entity 类）

**@去重检测**：
```bash
# 检测重复后缀 (1)、(2) 等
ls sources/xxx/*.md | grep -E '\ \([0-9]+\)$'
```

**@定锚 checklist（v37 新增）**：

-ingest 阶段二之前，必须完成定锚：

```
定锚 checklist（2分钟）：
1. 核心定义：这份资料的核心法律概念/罪名/制度是什么？
2. 常见误解：实践中这个概念最容易混淆/搞错的是什么？
3. 文档类型：司法解释/案例/书籍/修改决定？（对应不同处理策略）
```

### 阶段二：内容提取（知识编译）

**@Layered Confidence Rejection（v39 新增）**：内容提取时按置信度分层拦截，不同级别的内容走不同处理路径。

| 置信度 | 来源 | 处理方式 |
|--------|------|----------|
| 高 | 原文直接引用（逐字提取） | 直接写入 wiki |
| 中 | 原文转述（保留核心语义） | 写入 + 标注 〔存疑〕 |
| 低 | 个人解读/跨资料关联 | 写入 + 标注 〔待核实〕+ 同步 log.md |
| 极低 | 高度不确定/逻辑跳跃 | **拒绝写入 wiki**，只存入 log.md 待验证 |

**⚠️ 极低置信内容不得混入 wiki**：如果某条内容属于"高度不确定"或"逻辑跳跃"，即使很有价值也必须暂存 log.md，待验证后再决定是否写入 wiki。

**@核心原则**：分层提取，不是整文件处理

| 内容类型 | 处理单元 | 方式 |
|----------|----------|------|
| 法规/法条 | 逐条 | 每条独立释义 |
| 司法解释 | 逐条+逐款 | 解释条款是独立单元 |
| 案例分析 | 逐案 | 每案独立要旨 |
| 书籍章节 | 逐节/逐章 | 不能整本一起处理 |

**⚠️ 全部章节原则**：必须提取全部章节，不能只处理目录

**@差异驱动写入（v34 新增）**：

每份法律资料 ingest 时，必须在正文中包含：

```markdown
## 相比同类旧资料的核心差异
```

具体要求：

| 内容类型 | 差异提取重点 |
|----------|------------|
| 司法解释 | 必须提取"相对旧解释的具体变更点"（数额/主体/行为模式/程序要件），对照表格呈现 |
| 案例分析 | 必须提取"裁判要旨相比同类案件的差异化要点" |
| 法律法规修正 | 必须提取"相比修正前的主要变化点" |

示例（司法解释）：
```markdown
## 相比旧解释的核心变更

| 变更维度 | 旧解释（法释〔2016〕9号） | 新解释（法释〔2026〕6号二） |
|----------|--------------------------|----------------------------|
| 受贿罪数额标准 | 3万/20万/300万 | 维持不变 |
| 谋取利益要件 | 兜底条款 | 明确三种具体情形 |
```

**@相关主题自动发现**（NotebookLM 风格）：
```
内容提取完成后，自动扫描其他 wiki 页面的 tags 和 related 字段
→ 匹配相同/相似 tags 的页面
→ 在当前页面的 ## 相关 中添加建议链接
```

- 匹配当前页面 `tags` 中任意 tag 的其他 wiki 页面

**@主会话协调模式**：
```

Agent：数据提取（不直接写入）
```

## lint：健康检查

**@基础检查项**：

1. **断链检查**：`grep -r "\[\[" wiki/ | grep -v "\[\[.*\]\]"`
2. **矛盾检查**：同一数额标准在不同页面是否一致（如3万/20万/300万是否出现在所有相关罪名页）
3. **过时检查**：较新来源是否推翻旧结论
4. **空白检查**：重要概念是否已有页面
5. **交叉链接检查**：重要 concept 是否链接相关 entity/synthesis
6. **分类统计验证**：各目录文件数与预期一致
7. **源文件完整性验证**：wiki 页面的 source 字段指向的源文件是否存在

**@源文件完整性验证脚本**：
```bash
# 检查 wiki 页面的 source 是否仍存在于 sources/
missing=0
for f in wiki/summaries/*.md wiki/concepts/*.md; do
  if grep -q "^source:" "$f"; then
    # 提取 source 字段中的文件名
    sources=$(grep "^source:" "$f" | sed 's/^source: *//' | tr -d '[]' | tr ',' '\n')
    for src in $sources; do
      # 提取文件名（去除 [[ ]] 和路径前缀）
      fname=$(echo "$src" | sed 's/.*\///' | tr -d ']')
      if [ -n "$fname" ] && [ ! -f "sources/$fname" ] && [ ! -f "sources/网络文章/$fname" ]; then
        echo "⚠️ 源文件缺失：$fname（被 $f 引用）"
        missing=$((missing+1))
      fi
    done
  fi
done
[ "$missing" -eq 0 ] && echo "✅ 源文件完整性检查通过"
```

**@小型法律文件基线**（不触发空洞告警）：
| 类型 | 示例 | 预期大小 |
|------|------|----------|
| 宪法修正案 | 1988/1993/1999/2004/2018修正案 | 1-3KB |
| 法律修改决定 | 关于修改XX法的决定 | 1-3KB |
| 简单单行法 | 兵役法/人民调解法等 | 2-5KB |

**@进阶任务**：
- 概念空白检查：常见罪名是否都有 concept 页
- 发现待新建页面：旧知识是否被新资料推翻但页面未更新
- 建议下一步来源：根据空白建议补充
- 孤立页面检查：无入站链接的页面

## 常见错误警示

| 错误 | 后果 | 防范 |
|------|------|------|
| 省略第三档量刑 | 法律分析不完整 | 处理前先数有几档 |
| 整文件处理（不做分层） | 产出空洞框架 | 按处理单元逐条/逐案/逐节提取 |
| 生成占位符内容 | wiki 质量下降 | 永远只输出实际内容 |
| Agent 报告不可信 | 报告完成但实际遗漏 | 每批完成后立即用脚本验证 |
| 输出 < 500 字符未报警 | 内容空洞被放行 | 低于阈值必须立即复核 |
| 批量变更后未 commit | 大量变更遗留 | 批量删除/变更后立即 commit |
| 删除源文件 | 源资料灭失，不可逆 | **绝对禁止删除 sources/ 下任何文件** |
| ingest 后不建立链接 | wiki 页面孤立 | ingest 同时必须添加 wikilink |
| query 后不写回 wiki | 有价值发现消失在聊天记录 | 发现新分析/对比/连接时主动写入 |
| qmd embed 前未下载模型 | 进程卡在下载步骤 | 先 HF_ENDPOINT=... qmd embed 下载模型 |
| 跨批不完成阶段三/四 | 整合入库不完整 | 每批必须完成该批的阶段三+四 |
| qmd embed 等于整合入库 | 遗漏交叉链接等 | 整合入库 = 6步骤，不能用 embed 代替 |
| **编造法律条文内容** | 严重错误，误导用户 | **涉及数额/日期/适用范围等法条内容必须引用原文，不确定时主动向用户核验，严禁主观推测** |
| 法律条文对比未读原文 | 错误理解法律效力 | 涉及多部法律/解释比较时，必须先读取原文再下结论 |

## 批量 Ingest 执行规范（经验固化）

> 批量处理（≥10个文件）时，在阶段一至五基础上，额外遵循以下规则。

**@并行 Agent 策略（阶段二执行）**：

| 问题 | 根因 | 解决方案 |
|------|------|----------|
| Agent 报告"处理 3-5 个" | 上下文限制 | **立即用脚本验证** |
| Agent 报告"全部完成" | 未检查实际文件 | **精确比对** source vs target |
| 数量对不上 | 覆盖/遗漏 | **主会话接管处理遗漏** |

**@Phase门硬性检查（阶段一完成后执行）**：
```bash
SOURCE_COUNT=$(ls sources/xxx/*.md | wc -l)
TARGET_COUNT=$(ls wiki/dest/*.md | wc -l)
echo "Source: $SOURCE_COUNT, Target: $TARGET_COUNT"
[ "$SOURCE_COUNT" -eq "$TARGET_COUNT" ] || echo "⚠️ 数量不一致！"
```
数量核验失败 → 主会话接管 → 直接处理遗漏文件

**@批量规模适配索引策略（阶段四执行）**：

| 规模 | 索引策略 |
|------|----------|
| < 20 条 | 逐条列出 |
| ≥ 20 条 | **类目概述 + 目录 glob 引用**，避免 index.md 臃肿 |

> 阶段一/二/三详细步骤见对应章节。
