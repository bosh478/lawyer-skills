## @核心原则

**Karpathy 行为准则融合**：
- **假设透明**：ingest 前声明假设，不确定时主动询问，不要猜测
- **最小代码**：只做必要的变更，不做推测性功能
- **精准触及**：每个变更都可追溯到用户请求，阶段产出与请求直接对应
- **目标驱动**：每个阶段有明确的 verify 检查点

**RAG vs LLM Wiki**：ingest 不是"给文档建索引"，而是"把知识编译进网络"。

**绝对红线**：
- `sources/` 目录下的原始文件**只读不删**
- 永不生成占位符内容（`TODO`、`待补充`、`（略）`）
- 量刑档次三档/四档必须全部提取
- **法律准确性**：涉及具体法条内容（数额、日期、适用范围、罪名构成等）必须引用原文，禁止主观推测；**不确定时必须向用户核验**，严禁编造

## @速度优化（v28 新增）

**核心原则：不要等待，不要重复，不要二次整合**

### 1. 并行 Agent 快速验证流程

```
Agent 报告完成后 →
  立即执行验证脚本（不等待） →
  发现遗漏 → 主会话立即接管处理 →
  全部完成 → 立即进入阶段四
```

**验证命令（Agent 完成后立即执行）**：
```bash
# 精确比对：找出未处理的 source 文件
for f in sources/xxx/*.md; do
  name=$(basename "$f" .md)
  base_clean=$(echo "$name" | sed 's/（[0-9]*年[修正修订]*）//g')
  [ -f "wiki/concept_${name}.md" ] && continue
  [ -f "wiki/concept_${base_clean}.md" ] && continue
  echo "$name"
done
```

### 2. 内容写入优先原则

**❌ 旧做法（慢）**：
1. Agent 提取内容
2. 主会话读取旧文件
3. 合并/修改
4. 写入

**✅ 新做法（快）**：
1. Agent 直接写入 wiki 文件
2. 主会话只做验证
3. qmd update && embed

**阶段四整合入库流程（简化）**：
```bash
# 直接写入 + 验证，不做二次整合
qmd update && qmd embed
# log.md 更新（5秒内完成）
# 完成报告
```

### 3. 已存在文件检测（避免重复工作）

**启动 Agent 前必做**：
```bash
# 检查目标目录是否已有文件
ls wiki/concepts/xxx_*.md 2>/dev/null | wc -l
ls wiki/syntheses/xxx_*.md 2>/dev/null | wc -l

# 如果已有大量文件，说明可能已处理过
# 询问用户是否需要重新处理
```

### 4. 快速阶段一（精简版）

**文件数 ≤ 10 个时**：
1. `ls sources/xxx/*.md | wc -l`（5秒）
2. 启动 Agent（不逐个分析）
3. Agent 完成 → 验证 → 阶段四

**文件数 > 10 个时**：执行完整阶段一

### 5. Agent 任务分配优化

**不要让一个 Agent 处理太多内容**：
- 每个 Agent 处理的案例/章节数 ≤ 20
- 超过则拆分多个 Agent
- 4 个 Agent 并行 = 80 个案例/章节

## ingest：消化新资料

### 阶段一：快速扫描 + 方案确定（精简版）

**文件数 ≤ 10 个时（快速路径）**：
1. `ls sources/xxx/*.md | wc -l`
2. 判断文件类型和目标目录
3. 直接启动 Agent（不逐个分析）

**文件数 > 10 个时（完整路径）**：

#### 阶段一-0：查重筛选（前置过滤）

```bash
bash 查重筛选.sh <sources子目录名>/
```

**输出**：
- `已整理.md` — 已存在于 wiki 中的文件
- `未整理.md` — 需要继续处理的文件

#### 阶段一-A：快速扫描

1. **统计文件数**：`ls sources/xxx/*.md | wc -l`
2. **检测重复后缀**：`ls sources/xxx/*.md | grep -E '\([0-9]+\)$'`
3. **检查已有文件**：`ls wiki/concepts/xxx_*.md 2>/dev/null | wc -l`

#### 阶段一-B：方案确定

1. **判断处理模式**：快速路径（≤10个）还是完整路径（>10个）
2. **确定 Agent 数量**：按文件数/20 向上取整
3. **明确目标目录**：summaries / concepts / entities / syntheses

**阶段一产出**：处理方案（文件数、Agent数、目标目录）

---

### 阶段二：并行内容提取

**@核心原则**：分层提取，不是整文件处理

| 内容类型 | 处理单元 | 方式 |
|----------|----------|------|
| 法规/法条 | 逐条 | 每条独立释义 |
| 司法解释 | 逐条+逐款 | 解释条款是独立单元 |
| 案例分析 | 逐案 | 每案独立要旨 |
| 书籍章节 | 逐节/逐章 | 不能整本一起处理 |

**⚠️ 全部章节原则**：必须提取全部章节，不能只处理目录

**@提取细化级别**：

| 级别 | 说明 | 适用场景 |
|------|------|----------|
| **详细级（默认）** | 逐条逐款释义 + 关联规定 + 典型案例/裁判要旨 | 重要文件 |
| **穷尽级** | 逐字逐句解读 + 全部条款 + 全部关联 + 历史沿革 + 矛盾标注 | 极高精度要求 |

**穷尽级额外要求**：
- 逐字逐句逐款解读，不得遗漏任何表述
- 历史沿革：法条修改时间线、不同版本对比
- 矛盾标注：与其他法条/解释的冲突点明确标注
- 全部关联：所有相关法条、司法解释、指导案例逐一列出

**@并行 Agent 策略**：

1. **启动**：同步启动 N 个 Agent（N = 文件数/20 向上取整，max 4）
2. **监控**：Agent 完成后**立即**执行验证脚本
3. **补漏**：发现遗漏立即主会话接管
4. **继续**：全部完成后进入阶段三

**⚠️ Agent 报告不可信**：必须用脚本验证，不等待

---

### 阶段三：数量核验 + 内容质量检查

**@六步骤必须全部执行**：

1. **脚本精确比对**：`源文件数 == 目标文件数`
2. **文件大小检查**：< 3KB 可能为空（小型法律除外）
3. **内容抽检**：正文 > 500 字符、前100字符非空
4. **占位符扫描**：`待补充`、`见原文`、`（略）`
5. **量刑档次核对**：三档/四档不得遗漏
6. **数字/日期精确性核对**：涉及数额、日期、期限等关键数据必须与原文一致

**@快速核验命令**：
```bash
SOURCE_COUNT=$(ls sources/xxx/*.md | wc -l)
TARGET_COUNT=$(ls wiki/concepts/xxx_*.md 2>/dev/null | wc -l)
echo "Source: $SOURCE_COUNT, Target: $TARGET_COUNT"
[ "$SOURCE_COUNT" -eq "$TARGET_COUNT" ] || echo "⚠️ 数量不一致！"
```

**发现问题 → 立即修正，不进入阶段四**

## purge：根据原始文件删除对应 wiki 页面

**核心原则**：源文件是绝对红线——**只删除 wiki 页面，源文件永不删除**

**@执行流程**：
```
1. 查询 sources/INDEX.md，找到该源文件对应的所有 wiki 页面
2. 列出待删除的 wiki 页面，请求用户确认
3. 删除这些 wiki 页面
4. 从 sources/INDEX.md 中移除该源文件的记录
5. 更新 log.md（记录 purge 事件）
```

**⚠️ 不可逆操作，执行前必须二次确认**

## query：基于 wiki 回答问题

**@核心机制**：知识复合增长——查询过程中产生的有价值分析、对比和发现，存入 Wiki 成为新知识节点

**@回答规范**：
1. 多角度分析：对问题进行多维度拆解
2. 基于 wiki 回答：所有结论必须来自 wiki 资料
3. 引用规范：答案列出原文 + 来源（`[[页面名]]`格式）
4. 外部信息标注：若使用 vault 外资料，须特别说明

**@写回 wiki 触发条件**（强制执行）：
- 发现两个概念对比 → 创建 synthesis 对比页
- 发现类案规律 → 补充到相关 synthesis
- 发现矛盾 → 在相关页面标注
- 发现重要但 wiki 未覆盖的概念 → 新建 concept 页面
- 发现新旧解释更替关系 → 更新相关 concept 并标注

**@qmd 使用**：
```bash
qmd search "问题" -c tkk-vault-wiki --top-k 5    # BM25搜索（稳定）
qmd update                                       # 更新索引（ingest后必运行）
qmd embed                                       # 增量生成向量
```

## lint：健康检查

**@基础检查项**：

1. **断链检查**：`grep -r "\[\[" wiki/ | grep -v "\[\[.*\]\]"`
2. **矛盾检查**：同一数额标准在不同页面是否一致
3. **过时检查**：较新来源是否推翻旧结论
4. **空白检查**：重要概念是否已有页面
5. **交叉链接检查**：重要 concept 是否链接相关 entity/synthesis
6. **分类统计验证**：各目录文件数与预期一致
7. **源文件完整性验证**：wiki 页面的 source 字段指向的源文件是否存在

**@小型法律文件基线**（不触发空洞告警）：
| 类型 | 示例 | 预期大小 |
|------|------|----------|
| 宪法修正案 | 1988/1993/1999/2004/2018修正案 | 1-3KB |
| 法律修改决定 | 关于修改XX法的决定 | 1-3KB |
| 简单单行法 | 兵役法/人民调解法等 | 2-5KB |

**@lint 报告格式**：
```
## Lint 报告 [日期]

### 基础检查
- 断链：N 个
- 矛盾：N 个
- 占位符：N 个
- 空洞文件：N 个

### 统计
- concepts：N 个
- summaries：N 个
- qmd 索引：N documents

### 建议
- 建议补充来源：...
- 建议新建页面：...
```

## 输出格式标准

### summary（律师业务指引）

| 标准 | 要求 |
|------|------|
| 最低 | 关键要点 ≥ 5 条，每条原文提取 |
| 优秀 | 关键要点 ≥ 10 条，含程序步骤、风险点、合规要求，有 wikilink |

### concept（法律法规/法律原则）

| 标准 | 要求 |
|------|------|
| 最低 | 章节结构完整，每条编号+简要释义 |
| 优秀 | 原文引用+释义+关联规定，量刑档次完整（刑法类） |

### entity（司法解释/法条全文）

| 标准 | 要求 |
|------|------|
| 最低 | 解释要点 ≥ 3 条，涉及法条列表 |
| 优秀 | 每条原文+适用要点，有案例 wikilink |

### synthesis（案例分析）

| 标准 | 要求 |
|------|------|
| 最低 | 裁判要旨 ≥ 100 字（非"见原文"），基本信息完整 |
| 优秀 | 裁判理由+适用要点详细，有类案 wikilink |

## 常见错误警示

| 错误 | 后果 | 防范 |
|------|------|------|
| 省略第三档量刑 | 法律分析不完整 | 处理前先数有几档 |
| 整文件处理（不做分层） | 产出空洞框架 | 按处理单元逐条/逐案/逐节提取 |
| 生成占位符内容 | wiki 质量下降 | 永远只输出实际内容 |
| Agent 报告不可信 | 报告完成但实际遗漏 | **每批完成后立即用脚本验证** |
| 输出 < 500 字符未报警 | 内容空洞被放行 | 低于阈值必须立即复核 |
| 删除源文件 | 源资料灭失，不可逆 | **绝对禁止删除 sources/ 下任何文件** |
| ingest 后不建立链接 | wiki 页面孤立 | ingest 同时必须添加 wikilink |
| qmd embed 前未下载模型 | 进程卡在下载步骤 | 先 HF_ENDPOINT=... qmd embed 下载模型 |
| **编造法律条文内容** | 严重错误，误导用户 | **涉及数额/日期/适用范围等法条内容必须引用原文，不确定时主动向用户核验，严禁主观推测** |
