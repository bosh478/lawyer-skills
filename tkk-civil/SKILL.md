# 裁判文书分析 · tkk-civil v3.0

通用版 · 民事 / 行政 / 刑事 / 仲裁

同时输出：.docx 专业报告 + Obsidian Markdown 笔记

---

## 核心原则（铁律）

1. **原文优先**：所有案件信息、事实认定、证据内容、法律依据，必须原文引用文书中的表述，不得转述或概括走样
2. **不臆测**：文书未载明的信息（如当事人内心意图、庭外事实等），一律不得推测、假设或补充，必须明确标注"原文未载明"
3. **不修饰**：人物姓名、机构名称、日期、数字、金额等关键信息，必须与原文一致，不得简化、省略或"用XX代替"
4. **未载明不补全**：核心文件内容、合同条款、鉴定结论等，若判决书未详细摘录，必须标注"详见原判决书第X页"，不得自行填补
5. **分析归分析，事实归事实**：分析意见必须与判决书认定的事实明确区分，不得将分析意见混作事实陈述

**违反上述任何一条，即视为分析不合格，须重新修订。**

## 核心方法论

### 五步研读法（来源：美邦律师《复杂裁判文书快速研读攻略》）

| 步骤 | 内容 |
|------|------|
| 第一步 | 找到一审主要诉讼请求 |
| 第二步 | 分析一审主要诉讼请求（性质+类型）|
| 第三步 | 查找核心文件 |
| 第四步 | 找到法律依据（请求权基础）|
| 第五步 | 提炼裁判说理 |
| 第六步 | 做成案例概要 |

**分析时须同步运用请求权基础理论体系**，以"谁得向谁依据何种规范主张何种权利"为逻辑起点，对每一项诉讼请求逐一进行请求权检索。

## 工作流

### 第一步：读取文书

接收文件路径，确认格式（.docx / .pdf / .txt / .md / .doc），使用对应工具提取全文：
- `.docx` → `pandoc document.docx -t plain --wrap=none`
- `.pdf` → PDF skill 或 `pdftotext`
- `.txt` / `.md` → 直接读取
- `.doc` → 先使用 `C:/Users/汤康康/.claude/scripts/doc2md.py` 转换为 `.md` 再提取

### 第二步：文书类型识别与模块选择

阅读文书后，先完成前置判断：

#### ① 识别文书类型

标注文书类型：`民事判决书 / 行政判决书 / 刑事判决书 / 仲裁裁决书 / 裁定书 / 调解书 / 其他`

#### ② 根据文书类型启用模块

| 文书类型 | 启用模块 |
|---------|---------|
| 民事判决书 | 全部 10 个模块 |
| 行政判决书 | 全部 10 个模块（信息卡增加"被诉行政行为""复议机关"字段）|
| 刑事判决书 | 全部 10 个模块（信息卡改为"公诉机关/被告人/罪名/量刑"；争议焦点以"控方指控→辩方抗辩→法院认定"结构）|
| 仲裁裁决书 | 全部 10 个模块（信息卡增加"仲裁机构""仲裁条款""仲裁员"字段）|
| 裁定书 | 模块一（信息卡）+ 二（概述）+ 五（说理）+ 六（概要），省略争议焦点拆解、证据分析、精彩论述 |
| 调解书 | 模块一 + 二 + 六，省略争议焦点等，侧重分析调解方案的商业逻辑 |

#### ③ 识别审级

判断文书审级（一审/二审/再审）。如为**二审或再审**，必须启用所有"▶ 二审/再审附加要求"模块。

### 第三步：执行五步研读 + 进阶分析

#### 步骤一：找到一审主要诉讼请求

- 定位"原告诉称"或"原告上诉请求"部分
- 提取全部诉讼请求
- 识别主要诉讼请求（本金项，忽略仅主张利息的次要请求）

#### 步骤二：分析一审主要诉讼请求

**A. 诉讼法角度（诉的类型）**
- 给付之诉：请求对方履行特定行为
- 确认之诉：请求确认法律关系或权利存在与否
- 形成之诉（变更之诉）：请求变更或消灭法律关系

**B. 民商法角度（纠纷类型）**
- 合同纠纷：确认合同效力之诉 / 撤销合同之诉 / 解除合同之诉 / 继续履行合同之诉 / 违约损害赔偿之诉
- 侵权纠纷：侵权行为 + 侵害权利（利益）+ 损失及计算 + 责任因果关系 vs 损失因果关系 + 减轻/免责事由 + 双方过错
- 物权纠纷 / 身份法纠纷 / 其他

**C. 自行预判**（阅读结论前）

#### 步骤三：查找核心文件

- 合同纠纷：找出案涉合同名称及主要内容
- 侵权纠纷：找出侵权行为涉及的关键文件
- 其他：找出决定案件走向的关键文件

#### 步骤四：找到法律依据

**给付之诉** → 请求权基础法条：
- 合同请求权（民法典合同编）
- 侵权请求权（民法典侵权责任编）
- 无因管理请求权（民法典第121条）
- 不当得利请求权（民法典第122条）
- 物权请求权（民法典第235-237条）
- 人格权请求权（民法典第995条）

**确认之诉** → 确认法律关系效力的法条依据

**形成之诉** → 形成权的法条依据

#### 步骤五：提炼裁判说理（争议焦点深度拆解）

对每一个争议焦点，按以下结构展开：

```
【焦点N】：（一句话概括争议问题）
  ├─ 申请方主张：核心论点 + 法律依据 + 关键举证
  ├─ 被申请方抗辩：核心论点 + 法律依据 + 关键举证
  ├─ 法院认定：
  │   ① 援引的法律条文、司法解释或指导性案例
  │   ② 裁判推理链条（法律关系定性→请求权基础→
  │      构成要件逐一检验→事实认定→法律涵摄→结论）
  │   ③ 是否确立了新规则，或对既有规则作出了
  │      新的解释/限缩/扩张适用
  └─ 简评：该认定的合理性如何？有无值得商榷之处？
```

**▶ 二审/再审附加要求（强制）：**
每个争议焦点分析必须增加：
- **原审认定**：原审法院对该焦点的认定结论及核心理由
- **上诉/申请理由**：上诉人/申请人对原审认定的具体不服理由
- **本审认定**：本审法院的认定结论及理由
- **审级差异分析**：原审与本审在事实认定、法律适用或裁判思路上的关键分歧点是什么？

**▶ 再审案件特别要求：**
- **再审事由**：当事人据以申请再审的法定事由（新证据/原判适用法律错误/程序违法等）
- **再审审查范围**：法院全面审查还是仅围绕再审事由审查？
- **纠错深度**：仅纠正法律适用错误，还是重新认定了事实？对原生效判决的既判力有何影响？

#### 步骤六：做成案例概要

**A. 案件摘要**（3-5句话）：事实概要 + 争议焦点 + 判决结果 + 责任比例（如有）

**B. 法律关系图**（文字版）：
```
原告 →（合同关系）→ 被告
被告 →（侵权）→ 原告
```

**C. 裁判规则提炼**（如争议焦点有提炼价值）

## 进阶分析模块（模块七至十）

### 模块七：证据运用分析

- **关键证据清单**：列明证据名称、证据类型、待证事实
- **举证责任分配**：法院如何分配？是否涉及举证责任转移或倒置？是否进行了明确说理？
- **证据采信逻辑**：法院对争议证据的采信或排除理由是什么？有无值得关注的证据认定规则（电子证据采信标准、鉴定意见采纳规则、域外证据公证认证要求等）？

**▶ 二审/再审附加**：本审是否采纳了新证据？新证据对裁判结果的影响程度如何？

### 模块八：裁判要旨提炼

用 2-3 条规则性语言，概括本案确立或重申的裁判规则。

**格式**："在……情形下，……应当/不应当认定为……"

每条规则后**引用判决书中的关键原文表述**（而非段落编号），以便溯源。

二审改判案件，明确标注该规则是**本审新确立的**，还是**对原审规则的维持/修正**。

### 模块九：实务启示（分视角）

- **胜诉方视角**：哪些诉讼策略值得借鉴？
- **败诉方视角**：败因何在？如何避免或改进？
- **同类案件当事人视角**：对交易安排、合规管理或风险防范有何启示？
- **律师执业视角**：对代理策略、举证准备或法律论证方法有何参考价值？

**▶ 二审/再审附加视角**：
- **上诉策略复盘**：上诉人的上诉策略是否有效？如改判，关键翻盘点是什么？
- **原审代理反思**：原审败诉方的代理律师在一审阶段有哪些可以改进的地方？

### 模块十：裁判精彩论述摘录

摘录 2-3 段法院说理中逻辑最严密、论证最充分、或对实务最有指导意义的段落。

每段摘录后附**一句话点评**，说明其精彩之处（论证方法的创新性、规则阐释的清晰度、对后续案件的先例价值等）。

如为二审改判案件，**优先摘录本审与原审认定产生分歧的核心论述**。

## 批量处理模式（文件夹路径）

### 第一步：扫描文件夹

1. 使用 Glob 工具递归扫描文件夹及其所有子文件夹
2. 筛选支持的文书文件：`*.md`、`*.txt`、`*.pdf`、`*.docx`、`*.doc`
3. **按文件修改时间升序排列**（最早到最新）
4. 告知用户待处理文件数量及顺序

### 第二步：逐一处理（严格顺序）

对每一份文书：
1. 读取内容
2. 执行五步研读 + 进阶分析
3. 生成双格式输出（如未指定 `--docx` 或 `--md`）
4. 确认写入成功后再处理下一份
5. 每份处理完报告进度："已完成 N/M：[案件名称]"

### 第三步：严格隔离规则

> 以下规则**绝对不得违反**：

1. **信息不交叉**：每份文书分析仅包含该文书信息，绝不混入其他文书数据
2. **顺序不混乱**：严格按照清单顺序处理，先处理的结论不影响后处理的分析
3. **进度不跳跃**：处理完第N份前，不得读取或预览第N+1份的内容
4. **笔记不合并**：每份文书生成独立笔记文件，不得合并
5. **写入不覆盖**：若目标路径已存在同名文件，自动加 `-1`、`-2` 后缀区分

### 第四步：输出批次报告

```
本批次处理完成：
- 总计处理：N 份
- 成功：N 份
- 失败：0 份

文件清单：
1. [文件名1] → [Obsidian路径1] + [docx路径1]
2. [文件名2] → [Obsidian路径2] + [docx路径2]
```
