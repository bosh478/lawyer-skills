# 文章撰写 Skill

## 概述

本skill用于处理基于知识库资料撰写法律类文章的请求。核心流程包括：理解用户意图 → 检索Wiki知识库 → 设计文章框架 → 整合资料 → 输出初稿。

**知识库路径**：`D:\AI agent\tkk-library`
**输出路径**：`C:\Users\汤康康\Desktop\[文章标题].md`
**同步路径**：`D:\BaiduSyncdisk\第二大脑\[文章标题].md`

---

## 一、触发条件

当用户发送以下类型请求时调用本skill：

- "撰写一篇关于...的文章"
- "基于这份资料，写..."
- "帮我整理...的写作思路"
- "以...为蓝本，撰写..."
- "请根据...设计一个文章框架"
- 任何包含"撰写"、"写作"、"整理"且涉及知识库检索的任务

## 二、知识库检索

### 2.1 确定检索关键词

根据文章主题确定检索关键词，法律类文章常用检索词包括：

| 主题类型 | 检索关键词 |
|----------|------------|
| 受贿类 | 受贿、贿赂、贪污贿赂、职务犯罪、利用职务便利 |
| 贪污类 | 贪污、挪用公款、职务侵占 |
| 毒品类 | 贩卖毒品、非法持有毒品、运输毒品 |
| 诈骗类 | 诈骗、合同诈骗、集资诈骗 |
| 其他 | 刑事审判参考、入库案例、司法解释 |

### 2.2 检索优先级

检索时按以下优先级筛选：

1. **核心资料**：用户提供的基础资料（蓝本）
2. **司法解释**：法释〔2026〕6号（贪污贿赂解释二）等
3. **疑难问题**：王晓东《司法解释二疑难问题》等
4. **审判指导**：《职务犯罪审判指导》《刑事审判参考》
5. **入库案例**：人民法院案例库入库案例
6. **学术观点**：陈旭文讲座、张明楷著作等

### 2.3 检索方法

使用Grep工具搜索相关文件，再使用Read工具读取具体内容：

```
Grep: pattern="受贿|贿赂", path=D:\AI agent\tkk-library
Glob: pattern=**/*.md, path=D:\AI agent\tkk-library
```

## 四、写作要求

### 4.0 基本原则（必须遵守）

#### 4.0.1 数字、日期信息处理

**重要原则**：对文章中涉及的所有数字、日期、金额、比率、期限等信息，必须调取Wiki知识库（`D:\AI agent\tkk-library`）中的原始文件进行核实，不得仅凭记忆或此前对话中的引用进行处理。

具体要求：

- **金额类信息**（如受贿数额、犯罪金额）：必须查看原始案例文件，核实具体数字
- **日期类信息**（如判决日期、解释发布时间）：必须查看原始法律文件，确认准确日期
- **比率类信息**（如利率、比例）：必须查看原始规定，避免误用
- **期限类信息**（如追诉时效、刑期）：必须查看原始法条或司法解释
- **辑号、案号信息**（如《刑事审判参考》辑号、入库案例编号）：必须查看原始文件确认

处理流程：
1. 识别文章中涉及的数字、日期信息
2. 定位到Wiki知识库中的原始文件
3. 调取原始文件全文进行核实
4. 根据原始信息进行撰写

#### 4.0.2 法条内容处理

**重要原则**：对文章中涉及的所有法律条文，必须经过两步验证：第一步检查法条是否有效；第二步查看法条原文，不得随意解读。

具体要求：

**第一步：检查法条有效性**

- 查阅Wiki知识库中的**现行规范库**或**司法解释全书**
- 确认法条是否已被废止、修改或失效
- 注意法条的施行日期和失效日期
- 对于2026年后的案件，注意《刑法修正案（十二）》等新立法的适用范围

**第二步：查看法条原文**

- 必须调取法条原文内容，不得仅凭记忆或摘要引用
- 引用时应完整呈现法条核心内容
- 不得对法条进行随意解读或望文生义
- 对法条中关键术语的理解，应参考权威释义

处理流程：
1. 识别文章中涉及的法条（如刑法第385条、司法解释二第11条）
2. 检查法条是否有效（查阅现行规范库/司法解释全书）
3. 调取法条原文全文
4. 根据原文内容进行引用和解读
5. 注明法条来源和版本

---

### 4.1 案例案号要求

**所有案例必须标注案号来源**，禁止使用无出处的案例。

标注格式：
```
【案例名称】（案号/出处）
例如：
【赵强受贿案】（《刑事审判参考》第125辑1399号）
【翁某受贿案】（入库案例2024-18-1-404-001）
【卢某某受贿案】（《刑事审判参考》第138辑总第1578号）
```

常见案号格式：
- 《刑事审判参考》第X辑X号
- 入库案例20XX-XX-X-XXX-XXX
- 法院判决书：（年份）刑初/终XX号

### 4.2 表述要求

**自然语言优先原则**：

- 禁止在最终输出中使用表格
- 禁止使用列表（有序/无序）
- 将结构化内容改写为段落叙述

转换方式：
| 结构化内容 | 自然语言转换 |
|------------|--------------|
| 表格对比 | "A与B存在显著差异，A负责...，B负责..." |
| 列表要点 | "首先...其次...最后..."的递进句式 |
| 证据罗列 | "根据XXX证据，可以证明..." |

### 4.3 辩护要点要求

控辩对抗结构中，辩护要点必须**真正有利于被告人**，而非展示控方如何驳斥辩护方。

辩护方向：
- **无罪辩护**：质疑犯罪构成要件
- **罪轻辩护**：质疑数额认定、既遂/未遂、从犯
- **程序辩护**：质疑证据合法性、自首认定

## 五、输出流程

### 5.1 步骤一：理解用户意图

确认以下信息：

- 文章主题是什么
- 核心资料（蓝本）是什么
- 文章用途是什么（学术论文/实务指导/案例分析）
- 指定了哪种写作视角（控辩对抗/学术分析/案例分析）
- 有无其他特殊要求

### 5.2 步骤二：知识库检索

1. 根据主题确定检索关键词
2. 使用Grep/Glob检索相关文件
3. 读取核心资料和补充资料
4. 整理资料清单

### 5.3 步骤三：设计文章框架

根据用户指定的用途和视角，选择合适的写作框架，或设计定制化框架。

### 5.4 步骤四：撰写初稿

1. 按照框架组织内容
2. 将核心资料与知识库补充资料有机结合
3. 为所有案例标注案号来源
4. 按照自然语言要求调整表述

### 5.5 步骤五：结论部分

结论部分应包含：

1. **总体特征**：该类案件的共性特征和司法难点
2. **处理原则**：如罪刑法定、主客观相一致、有利于被告人、程序正义
3. **未来展望**（可选）：如对司法实践的展望、对辩护策略的建议

### 5.6 步骤六：输出与同步

1. 保存文件至桌面
2. 同步至Obsidian vault
3. 向用户确认完成
